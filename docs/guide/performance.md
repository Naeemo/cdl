# 性能优化指南

## 性能基准

CDL 编译性能目标：
- **小型图表** (< 10 行数据): < 10ms
- **中型图表** (< 100 行数据): < 50ms
- **大型图表** (< 1000 行数据): < 100ms

## 影响因素

1. **数据量** - 数据行数直接影响解析和转换时间
2. **图表复杂度** - combo、sankey 等复杂图表渲染开销更大
3. **交互配置** - 大量交互规则增加处理时间
4. **数据源类型** - SQL/API 查询时间取决于外部服务

## 优化建议

### 数据层面

✅ **限制数据行数**
- 建议 < 1000 行用于实时渲染
- 大数据集使用分页或聚合

✅ **预聚合**
- 在数据库层面完成聚合，减少前端处理
- 使用 `GROUP BY` 或窗口函数

✅ **字段精简**
- 只选择图表实际需要的字段
- 避免 `SELECT *`

### 语法层面

✅ **避免过度嵌套**
- 简单图表比嵌套 combo 更快
- 复杂需求考虑多个图表联动

✅ **合理使用缓存**
```cdl
@source('sales_db')
@cache(300)  // 缓存 5 分钟
```

✅ **控制图表数量**
- 单页面渲染 < 10 个复杂图表
- 使用分页或懒加载

### 渲染层面

✅ **启用响应式**
- 避免频繁 `setOption`，使用 `resize` 代替重绘
- `@responsive true` 优化重绘逻辑

✅ **简化样式**
- 过多的动画效果会降低性能
- 大数据集考虑关闭动画：`@animation "none"`

✅ **按需渲染**
- 图表在可视区域才渲染
- 使用 `IntersectionObserver` 或虚拟滚动

## 性能测试

运行内置性能基准：

```bash
node packages/compiler/dist/benchmark.js
```

或使用集成测试中的性能检查：

```bash
node packages/integration-test/test-v06.js
```

## 常见性能问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 编译慢 | 数据行数过多 | 聚合或分页 |
| 渲染卡顿 | 图表太复杂 | 简化图表类型 |
| 交互延迟 | zoom/brush 数据量大 | 限制数据点 < 10000 |
| 内存泄漏 | 未销毁图表实例 | 调用 `chart.dispose()` |

## 监控指标

建议监控：
- 编译时间（`console.time()`）
- 渲染时间（ECharts `setOption` 耗时）
- 内存使用（大数据集注意内存泄漏）
- FPS（动画流畅度）

## 未来优化方向

- 增量编译（只重新编译修改部分）
- Web Worker 后台编译
- 虚拟化渲染（大数据集只渲染可视区域）
- 服务端渲染（SSR）缓存
